# 设计文档同步规范

> **制定人**：PM Agent
> **生效日期**：2026-04-03
> **适用范围**：所有实现层 Agent（Infra / Services / Core / Tools / TUI / CLI）

---

## 背景

代码评审阶段发现，各 Agent 在实现过程中对接口、字段、签名做了调整，但均未同步更新对应的设计文档，导致代码与文档产生偏差，增加了后续维护和交接成本。本规范明确文档同步的触发条件、执行方式和验证要求。

---

## 核心原则

**设计文档是代码的"合同"，代码修改若导致合同变化，必须同步更新合同。**

---

## 触发条件

任何以下情况发生时，负责该层的 Agent **必须**在同一 PR/任务中更新对应设计文档：

| 变更类型 | 示例 | 需更新文档 |
|---------|------|-----------|
| 接口签名变更 | 方法参数增减、返回值类型变化 | ✅ 必须 |
| 新增公开类型/函数 | 新增 struct、interface、exported func | ✅ 必须 |
| 删除或重命名公开符号 | 删除接口方法、重命名 struct 字段 | ✅ 必须 |
| 模块职责变化 | 某模块承担了设计中未描述的额外职责 | ✅ 必须 |
| 依赖关系变化 | 新增跨层依赖、移除已有依赖 | ✅ 必须 |
| 内部实现优化 | 仅优化算法、不改变外部行为 | ❌ 可选 |
| 注释/文档注解修改 | 不影响接口和行为 | ❌ 可选 |

---

## 执行步骤

每次代码变更完成后，Agent 应按以下步骤自查：

### Step 1 — 对照设计文档检查

阅读本层设计文档（`docs/project/design/<layer>.md`），逐项核对：

```
□ 所有公开接口/方法签名是否与文档一致？
□ 所有公开 struct/type 字段是否与文档一致？
□ 模块职责描述是否仍然准确？
□ 模块间依赖关系是否仍然准确？
□ 有无新增的关键设计决策未被记录？
```

### Step 2 — 更新设计文档

若发现差异，在设计文档中：

1. **更新对应章节**的接口定义、字段描述、流程图等
2. **在文档头部更新版本号**（v1.0 → v1.1）和修改日期
3. **在文档末尾追加变更记录**（见格式要求）

### Step 3 — 更新文档版本记录

在设计文档末尾维护以下变更日志区块：

```markdown
---

## 变更记录

| 版本 | 日期 | 修改人 | 变更摘要 |
|------|------|--------|---------|
| v1.1 | YYYY-MM-DD | Agent-<Layer> | 修改 XxxInterface.Method 签名：新增 ctx 参数 |
| v1.0 | YYYY-MM-DD | Agent-<Layer> | 初稿 |
```

---

## Tech Lead 评审要求

Tech Lead 在进行代码评审时，**必须**将设计文档符合度作为独立评审维度：

- 评审报告中必须包含 `## Design vs Implementation Delta` 章节
- 若发现代码与文档存在未记录的偏差，应列为至少 **P1** 问题
- 若偏差涉及接口契约破坏（影响跨层调用），应列为 **P0** 问题

---

## QA 验收要求

QA 在验收各层时，需额外检查：

```
□ 设计文档版本号是否高于 v1.0（有无更新记录）？
□ 设计文档中的接口定义与实际代码是否一致（抽样检查 3 个关键接口）？
□ 若文档未更新但代码明显偏离设计，应记为 P1 问题
```

## 后续行动

本规范生效后，各 Agent 需针对**当前已发现的设计文档偏差**（见各层代码评审报告 `Design vs Implementation Delta` 章节）补充完成一次文档补齐：

1. 每层 Agent 阅读本层的代码评审报告（`docs/project/reviews/code-review-<layer>.md`）
2. 针对 Delta 章节中列出的偏差逐一更新设计文档
3. PM 在下次巡检时确认各层设计文档已更新至 v1.1+
