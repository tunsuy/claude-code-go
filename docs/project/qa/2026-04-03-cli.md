## 3. 单元测试质量评估

### 3.1 `internal/commands` — 质量良好 ✅

**测试文件**：`registry_test.go`，包含 5 个测试用例，覆盖 `Lookup`、`CompletePrefix`、`All`、`Execute`、`Help` 五个核心场景。

**优点**：
- 使用黑盒模式（`package commands_test`），隔离性良好。
- `TestCommandExecute` 直接调用 `cmd.Execute(ctx, "")` 并断言 `result.ShouldExit`，是良好的行为语义测试。
- `TestHelpCommand` 验证 `/help` 返回非空文本，覆盖了注册表聚合逻辑。
- `TestRegistryCompletePrefix` 验证了补全前缀逻辑的核心路径。

**不足**：
- **无表驱动测试**：`cmdVim`（开/关状态两分支）、`cmdTheme`（空参数 vs 有参数）、`cmdModel`（查询 vs 设置）等命令有多分支，均未用表驱动覆盖，导致 `builtins.go` 中多个函数仅 20%~50% 覆盖率。
- `CompletePrefix` 未测试边界场景：空前缀返回全集、无匹配返回空列表、前缀精确匹配 vs 子串匹配。
- `cmdClear`、`cmdCompact`、`cmdReview` 等命令行为逻辑（如 `NewMessages` 字段设置）未被验证。

### 3.2 `internal/bootstrap` — 质量偏弱 ⚠️

**测试文件**：`bootstrap_test.go`，4 个测试函数。

**优点**：
- `TestMain` 正确隔离环境（重定向 `HOME`、`XDG_CONFIG_HOME`，清除 API key 环境变量），体现良好的测试环境意识。
- `TestBuildContainer_MissingConfig_Succeeds` 验证了"无配置文件时不崩溃"这一重要容错场景。
- `TestResolveModel_Override` 是干净的纯函数单元测试，输入/期望/断言明确。
- `TestCollectHeadlessPrompt_FromArgs` 使用 `t.Fatalf` 快速失败，断言风格符合规范。

**不足**：
- **无表驱动测试**：`resolveModel` 有 3 条分支（override/settings/default），仅测了第一条；`isPrintMode` 有多种参数组合，完全未测；`collectHeadlessPrompt` 有两条路径，仅测了 args 路径。
- **关键路径零覆盖**：`runMCPServe`（MCP 握手）、`buildRootCmd`（headless 分支数量验证）、`runInteractiveOrHeadless`（主分发逻辑）、所有输出消费函数均无任何测试。
- **Mock 策略缺失**：`headlessRun` 依赖真实 `QueryEngine`，未为其注入假实现。测试策略文档要求 interface-based mock，但此处未落实。
- **`defer` + `os.Exit` 问题**（见 P2-2）：`TestMain` 中 `defer os.RemoveAll(tmpHome)` 因 `os.Exit(m.Run())` 而不被执行，临时目录泄漏。

### 3.3 `cmd/claude` — 最小合理覆盖 ✅/⚠️

**测试文件**：`main_test.go`，3 个测试用例。

**优点**：
- `TestMain_VersionFlag` 直接断言 `HandleFastPath` 返回值，简洁有效。
- `TestMain_HelpFlag` 通过 `recover` 验证 `bootstrap.Run("--help")` 无 panic，是对 cobra 集成的有效 smoke test。
- 包级覆盖率 0.0% 属统计工具限制（`main()` 含 `os.Exit`），非实际缺失。

**不足**：
- `TestMain` 中 `defer func() { os.Args = origArgs }()` 同样因 `os.Exit` 而不被执行（与 `bootstrap_test.go` 同类问题）。
- 仅测试 `--version`，未测试 `-v` 别名、`--` 终止符后的快速路径不触发、无参数时快速路径不触发等其他场景。

### P1（应尽快修复）

#### P1-1：`internal/bootstrap` 测试覆盖率严重不足（19.6%，目标 ≥60%）

**位置**：`internal/bootstrap/` 整体

**现象**：核心启动路径 `Run`、`isPrintMode`、`buildRootCmd`、`runInteractiveOrHeadless`、`runMCPServe`、所有输出消费函数（`consumeText`/`consumeStreamJSON`/`consumeJSON`）、`applyPermissionFlags` 覆盖率均为 0%。三包联合覆盖率 49.2%，低于测试策略 ≥60% 目标约 11 个百分点。

**影响**：关键路径回归风险极高，任何启动逻辑修改均无测试保护网。

**修复建议**：
1. 补充 `isPrintMode` 的表驱动测试（`-p`、`--print`、`--model -p` 误判、`--` 终止符后不判断等排列）。
2. 补充 `buildRootCmd(headless=true/false)` 测试，断言子命令注册数量差异。
3. 补充 `runMCPServe` 的 initialize 握手测试（使用 `os.Pipe()` 模拟 stdin/stdout）。
4. 补充 `collectHeadlessPrompt` stdin 路径测试（用 `strings.NewReader` 替换 `os.Stdin`——需重构为接受 `io.Reader` 参数）。
5. 补充 `applyPermissionFlags` 各权限模式的单元测试。
6. 补充 `consumeText`/`consumeStreamJSON`/`consumeJSON` 的输出格式测试（用 channel 驱动假 engine 消息）。

#### P1-2：`rootCmd` 包级全局变量破坏测试隔离（复核 Tech Lead P1-A）

**位置**：`internal/bootstrap/bootstrap.go:28`

```go
var rootCmd *cobra.Command   // 包级全局变量

func Run(args []string) error {
    rootCmd = buildRootCmd(headless)  // 每次调用覆写全局
    ...
}
```

**现象**：若两个 goroutine 并发调用 `Run()`（如 `t.Parallel()` 测试中），会对同一包级变量产生写竞争，`go test -race` 将报告 data race。当前测试因 `Run()` 覆盖率为 0% 而未触发，属隐蔽隐患。

**影响**：违反设计 §7.3 "避免全局单例"原则；一旦补充 `Run()` 测试，race 将立即暴露。

**修复建议**：将 `rootCmd` 降级为 `Run()` 函数内的局部变量，`Execute()` 兼容函数直接调用 `Run(os.Args)` 即可。

---

#### P1-3：`isPrintMode` 与 cobra 参数解析存在逻辑二义性（复核 Tech Lead P1-B）

**位置**：`internal/bootstrap/bootstrap.go:69-79`

**现象**：`isPrintMode` 不感知其他 flag 的参数消耗。`claude --model -p` 中 `-p` 是 `--model` 的值，但会被误判为 print 模式，导致全部子命令注册被跳过。`claude somesubcmd -p` 同理。

**影响**：在特定参数组合下整个命令树启动模式被误判，子命令被错误跳过注册，用户报告行为异常。

**修复建议**：至少添加针对 `claude --model -p` 场景的单元测试，在函数注释中文档化已知二义性边界（即使暂时允许误判，也需显式记录）。

#### P1-4：`--resume`/`--continue` flag 解析后静默丢弃，对话恢复功能为零（复核 Tech Lead P1-C）

**位置**：`internal/bootstrap/root.go:168-214`

```go
// runInteractive 调用处
return runInteractive(f, opts)  // f.continueSession / f.resume 传入但被忽略

func runInteractive(f *rootFlags, opts ContainerOptions) error {
    ...
    m := tui.New(container.QueryEngine, container.AppStateStore, false, true)
    // f.continueSession / f.resume 从未使用
}
```

**现象**：`f.continueSession`（`-c`）和 `f.resume`（`-r`）在 `runInteractive` 中被完全忽略，未传递给 TUI 或 `AppStateStore`。

**影响**：设计 §1 核心功能点（`-c` 恢复最近会话、`-r` 恢复指定会话）完全无效，用户可见的功能性空洞。

**修复建议**：在 `runInteractive` 中将这两个 flag 写入 `AppStateStore`（如 `store.SetResumeTarget(f.resume, f.continueSession)`），或至少添加 `// TODO(P1):` 注释标记。

#### P1-5：`mcp serve` 的 `CLAUDE_CODE_ENTRYPOINT` 缺失 + serve 为占位骨架（复核 Tech Lead P1-D/P1-E）

**位置**：`internal/bootstrap/mcp.go:45-78`

**现象一（entrypoint 缺失）**：进入 `mcp serve` 路径时未通过 `AppStateStore.SetEntrypoint("mcp")` 设置 entrypoint 标识。设计 §4.3 和 §7.6 明确要求，下游工具层权限检查依赖此变量做行为差异化。

**现象二（serve 骨架不完整）**：
```go
switch req.Method {
case "initialize":
    // 仅返回握手响应
default:
    // 所有其他方法返回 -32601 Method not found
}
```
缺失：
- `tools/list` → MCP 客户端无法发现可用工具
- `tools/call` → 工具调用完全无法转发
- `enc.Encode` 错误被 `//nolint:errcheck` 静默，写入 stdout 失败时服务端无感知
- `serverInfo.version` 硬编码 `"0.1.0"`，未读取 `appVersion`

**影响**：MCP 模式仅能完成协议握手，无法被真实 MCP 客户端使用工具。

#### P1-6：`wire.go` 缺少设计要求的编译期接口断言（复核 Tech Lead P1-F）

**修复建议**：在 `wire.go` 中以 `// TODO:` 形式添加意图占位断言，或在对应包接口稳定后立即补充。

### P2（建议优化）

#### P2-1：`plugin` 缺少 `plugins` 别名，`update` 缺少 `upgrade` 别名

**位置**：`internal/bootstrap/plugin.go:10`、`internal/bootstrap/misc.go:42`

设计 §1 命令树明确标注 `plugin (alias: plugins)` 和 `update (alias: upgrade)`。Cobra 通过 `Aliases: []string{"plugins"}` 字段一行即可修复，建议同步添加对应测试用例（调用 `rootCmd.Find("plugins")` 断言非 nil）。

#### P2-2：`TestMain` 中 `defer os.RemoveAll` + `defer os.Args 还原` 因 `os.Exit` 不执行

**位置**：`internal/bootstrap/bootstrap_test.go:13-27`、`cmd/claude/main_test.go:12-17`

```go
// bootstrap_test.go
tmpHome, _ := os.MkdirTemp(...)
defer os.RemoveAll(tmpHome)  // ← 不执行，目录泄漏
os.Exit(m.Run())             // ← os.Exit 绕过所有 defer

// main_test.go
defer func() { os.Args = origArgs }()  // ← 不执行
os.Exit(m.Run())
```

**修复建议**（两处同类问题）：
```go
code := m.Run()
os.RemoveAll(tmpHome)   // 显式清理，无需 defer
os.Exit(code)
```

#### P2-3：`runInteractive` 中 SIGINT goroutine 接收信号后无任何动作

**位置**：`internal/bootstrap/root.go:204-211`

```go
go func() {
    <-sigCh
    // 收到信号后什么也不做
}()
```

`signal.Notify(sigCh, ...)` 拦截了操作系统的默认终止行为，但 goroutine 什么也不做，反而可能延迟进程对 SIGINT 的响应。

**修复建议**：要么删除此代码段（让 BubbleTea 完全管理信号），要么实现实际清理逻辑（调用 `p.Quit()` 或 cancel context）。

---

#### P2-4：`openBrowser` 失败时无用户提示

**位置**：`internal/bootstrap/auth.go:119`

```go
openBrowser(authURL) //nolint:errcheck
```

在无显示器服务器上运行时，浏览器打开失败但用户无任何提示，只能看到 URL 输出到 stdout。

**修复建议**：显式处理错误并打印提示（如"浏览器打开失败，请手动访问以上 URL"），而非静默丢弃。

---

#### P2-5：`BuildHeadlessContainer` 为 `BuildContainer` 直接 passthrough，缺具体 TODO

**位置**：`internal/bootstrap/wire.go:101-107`

当前完全等价于 `BuildContainer`。若后者未来引入 TUI 相关初始化（如 terminal capability 探测），headless 路径将被误带入。

**修复建议**：添加具体的 `// TODO: skip MCP pool init in headless mode` 等待办，防止遗忘。

---

#### P2-6：`consumeText` 中 `MsgTypeTurnComplete` 无条件追加换行，可能产生冗余空行

**位置**：`internal/bootstrap/run.go:117-119`

```go
case engine.MsgTypeTurnComplete:
    fmt.Fprintln(w) // 无条件追加 \n
```

LLM 响应本身以 `\n` 结尾时（非常常见），会额外追加一个空行，导致 `-p` 模式 pipeline 中出现冗余换行。

**修复建议**：追踪最后写入字节，仅在响应非 `\n` 结尾时补充换行。

---

#### P2-7：`resolveAPIKey` 使用 `context.Background()` 无超时控制

**位置**：`internal/bootstrap/wire.go:126`

`CheckAndRefreshIfNeeded(context.Background())` 无超时，网络异常时启动可能无限期阻塞。

**修复建议**：注入带超时的 context（如 5 秒），并在函数注释中说明此调用存在网络延迟。

---

#### P2-8：`internal/commands` 缺乏表驱动测试，多命令分支未覆盖

**位置**：`internal/commands/builtins.go`、`registry_test.go`

以下命令有双分支但仅覆盖一条：`cmdVim`（开/关状态）、`cmdTheme`（空参数/有参数）、`cmdModel`（查询/设置）、`cmdClear`（`NewMessages` 是否设置）。`CompletePrefix` 未测试空前缀、无匹配等边界。

**修复建议**：改为表驱动写法，预计可将 `internal/commands` 覆盖率从 68.1% 提升至 ≥85%。
