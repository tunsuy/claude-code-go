## 3. 单元测试质量评估

### 3.1 `internal/config`

**优点**：
- 采用黑盒测试（`package config_test`），符合测试策略规范。
- 覆盖了分层合并的核心场景：三层叠加、Policy 覆盖 Local、环境变量优先级、`uniqueAppend` 去重、带类型 Hooks 的轮转。
- `TestLoad_EnvVarOverridesFile` 正确使用 `t.Setenv()`，测试结束后自动还原，无全局副作用。

**缺陷**：
- **未使用表驱动测试**：7 个测试函数均为独立函数，违反测试策略"优先使用 table-driven 写法"的要求；同类场景（如多种 Policy 覆盖组合）缺乏参数化覆盖。
- **缺少错误路径**：未测试 JSON 格式错误的 settings 文件（`loader.Load()` 的错误分支覆盖为零）。
- **缺少 bool 指针合并场景**：`CleanupPeriodDays`、`RespectGitignore` 等 `*bool` 字段的三层合并行为未经测试。
- **Hooks 追加语义无测试**：`applyLayer` 对 Hooks 执行 `append` 而非替换，与 `applyPolicyOverrides` 的替换语义不一致，现有测试均未验证 Local 层能否覆盖 User 层 Hook（对应 CR P1-2）。

### 3.2 `internal/session`

**优点**：
- 覆盖 4 个关键路径：空文件读取、腐坏行跳过（无错误返回）、Raw 字节深拷贝（缓冲区安全）、New/Resume 完整生命周期。
- `TestSessionStore_AppendAndReadAll` 显式修改 `Raw` 副本并验证原始不变，充分验证了 buffer-reuse 修复（RC-5）。
- `TestResume_MissingSession` 验证了文件不存在时的错误返回。
- 黑盒测试（`package session_test`）。

**缺陷**：
- **无并发压力测试**：`SessionStore.AppendEntry` 内部有 `sync.Mutex`，但未有任何并发写入测试（goroutine 并发 `AppendEntry`）验证锁正确性。
- **无超大行测试**：未验证单行超过 4 MiB 缓冲区时 `scanner.Scan()` 的行为（`bufio.Scanner.Buffer` 硬限制）。
- **无 double-Close 测试**：`Close()` 后再次调用的行为未经验证。

### 3.3 `internal/permissions`

**优点**：
- 白盒测试（`package permissions`），`stubPermTool` 和 `stubRegistry` 实现了完整接口，测试隔离良好。
- 覆盖权限流水线的主要分支：bypass 模式、alwaysDeny、alwaysAllow、alwaysAsk、dontAsk 模式、plan 模式（读允许/写拒绝）、否决计数追踪、Ask 通道 nil 时降级为拒绝、用户审批后通过。
- 表驱动测试结构较好，边界用例充分。

**缺陷**：
- **`PermissionModeAcceptEdits` 分支无覆盖**：checker 中存在 `acceptEdits` 模式分支，所有测试均未触发该路径。
- **context 取消场景缺失**：`RequestPermission` 等待用户审批期间如果 context 被取消，行为未经测试。
- **`matchRules` 多来源场景**：仅测试了单来源规则；多个 `PermissionRule` 叠加时的优先级行为未覆盖。
- **`DenialTrackingState.RecentDenials` 无上限**：记录层无最大条数限制，高频拒绝场景下内存持续增长，无对应测试验证或防护。

### 3.4 `internal/bootstrap`

**优点**：
- `TestMain` 正确隔离 HOME/XDG_CONFIG_HOME 至临时目录，防止测试污染真实配置文件。
- `TestBuildContainer_MissingConfig_Succeeds` 验证了无配置文件时构建容器不崩溃（空 API Key 可接受）。
- `TestResolveModel_Override` 和 `TestCollectHeadlessPrompt_FromArgs` 覆盖了两个纯函数的基本路径。

**缺陷**（严重）：
- **`HandleFastPath` 完全无测试**：`--version`/`-v` 快速路径是启动最先执行的分支，完全无覆盖。
- **`isPrintMode` 完全无测试**：原始 args 扫描逻辑无任何验证。
- **`applyPermissionFlags` 完全无测试**：权限标志（`--dangerously-skip-permissions`、`--permission-mode`、`--add-dir`）应用到 AppStateStore 的逻辑完全无覆盖。
- **`BuildHeadlessContainer` 无独立测试**：虽目前委托给 `BuildContainer`，但 headless 路径差异（如未来 MCP 跳过）无测试保护。
- **`runDoctor` / subcommands 无测试**：`doctor`、`config`、`mcp` 等子命令的输出逻辑无任何测试。

### P1（应尽快修复）

#### P1-QA-1 · `applyEnvOverrides` 不可依赖注入，并发测试存在全局副作用风险

**对应**: CR P1-3（未修复）
**文件**: `internal/config/loader.go`

`applyEnvOverrides` 直接调用 `os.Getenv`。当前测试使用 `t.Setenv` 绕过，顺序执行时可接受，但若将来测试并行化（`t.Parallel()`），`t.Setenv` 会与其他并行测试互斥，实质上阻止了 config 测试并行化。

**修复方案**: 在 `Loader` 结构体中增加 `environ func(string) string` 字段，默认为 `os.Getenv`，并提供 `NewLoaderWithEnv` 变体用于测试注入。

---

#### P1-QA-2 · `testutil` 包过于稀少，无法支撑基础层测试复用

**对应**: CR P1-4（未修复）
**文件**: `pkg/testutil/testutil.go`

当前 `testutil` 仅有 2 个函数（`AssertNoError` / `AssertError`），且 `session/store_test.go` 未使用它（直接调用 `t.Fatalf`）。各测试文件中大量重复 `t.TempDir()` + 文件写入 boilerplate。

**修复方案**: 补充 `MustTempDir(t)`、`AssertEqual[T comparable](t, got, want T)`、`MustWriteFile(t, path, content)` 等基础工具函数。

---

#### P1-QA-3 · `internal/bootstrap` 测试覆盖率 19.6%，关键路径无测试保护

**文件**: `internal/bootstrap/bootstrap_test.go`

覆盖率 19.6% 为四包中最低，且以下关键函数完全无测试：
- `HandleFastPath(args []string)` — `--version`/`-v` 快速路径
- `isPrintMode(args []string)` — 原始 args 扫描（headless 模式判断依赖此函数）
- `applyPermissionFlags(container, f)` — CLI 权限标志 → AppState 的最终落点，覆盖三条独立路径：`--dangerously-skip-permissions`、`--permission-mode`、`--add-dir`
- `buildRootCmd(headless bool)` — headless 模式下子命令集合的差异未经测试
- `BuildHeadlessContainer` — 无独立测试，仅因委托给 `BuildContainer` 才间接触达

任一函数出现回归均无测试保护。

**建议最低补充**：
1. `TestHandleFastPath` — 表驱动；`--version`/`-v` 返回 `true`，其余参数返回 `false`。
2. `TestIsPrintMode` — 表驱动；`-p`/`--print` 返回 `true`，`--` 截止后的参数不计入。
3. `TestApplyPermissionFlags` — 分三个子用例验证三条标志路径，通过 `BuildContainer` + `applyPermissionFlags` 闭环测试。

### P2（建议优化）

#### P2-QA-1 · `internal/config` 测试未使用表驱动写法

**文件**: `internal/config/loader_test.go`

7 个测试函数均为独立函数，无法参数化扩展。例如 Policy 覆盖场景（APIKey、Model、BaseURL、Allow/Deny 各字段）可统一为一个表驱动测试，减少约 50% 的重复代码，同时更易于补充新场景。

---

#### P2-QA-2 · `applyPolicyOverrides` Deny 规则前置顺序无专项测试

**文件**: `internal/config/loader.go`，`applyPolicyOverrides` 函数

Policy 的 Deny 规则必须前置于用户 Deny 规则（安全语义）。现有测试 `TestLoad_PolicyOverridesLocal` 仅验证字段值被覆盖，未验证 Deny 规则在合并后切片中的顺序。这是一个安全相关的不变量，应有专项测试。

**建议**：补充 `TestLoad_PolicyDenyRulesPrependedBeforeUserRules`，同时配置用户层和 Policy 层各含一条 Deny 规则，断言 Policy 规则位于合并结果切片的前端。

---

#### P2-QA-3 · `SessionStore.AppendEntry` 无并发写入压力测试

**文件**: `internal/session/store_test.go`

`AppendEntry` 内部使用 `sync.Mutex`，但无任何并发写入测试。建议补充 goroutine 并发 `AppendEntry` 测试（`-race` 会捕获实际数据竞争），以防止未来锁被错误简化。

---

#### P2-QA-4 · `PermissionModeAcceptEdits` 分支无测试

**文件**: `internal/permissions/checker_test.go`

权限检查器支持 `acceptEdits` 模式（`CanUseTool` 第 7 步），但 17 个测试函数中无一覆盖该分支。建议补充：在 `AcceptEdits` 模式下文件编辑类工具自动批准的用例。

---

#### P2-QA-5 · `DenialTrackingState.RecentDenials` 无上限防护

**文件**: `internal/permissions/denial.go`

`RecentDenials` 是无界切片，高频拒绝场景（如被锁死的批量工具调用）可导致内存持续增长。建议在设计文档中明确上限（如最近 100 次），并补充超限截断测试。

---

#### P2-QA-6 · `internal/config/settings.go` 为空文件（CR P2-5 未解决）

**文件**: `internal/config/settings.go`

该文件仅有 package 声明，`SettingsJson` 及相关类型均在 `loader.go` 中定义。文件名与内容矛盾，在 Git 历史中制造噪音。建议将 `SettingsJson` 及常量迁移至该文件，或直接删除。

---

#### P2-QA-7 · `internal/permissions/` 未出现在设计文档包列表中

**参考**: `docs/project/design/infra.md`

`infra.md` 列出的包为 `pkg/types/`、`pkg/utils/`、`internal/config/`、`internal/state/`、`internal/session/`、`internal/bootstrap/`。`internal/permissions/` 为实现中新增的包，未在设计文档中描述。

**建议 PM 触发 Tech Lead 补充评审**，确认 `internal/permissions/` 是否为基础设施层的正式组成部分，并更新 `infra.md`。
