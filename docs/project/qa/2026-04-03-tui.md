### P1（应尽快修复）

#### P1-A：滚动视口未按 `termHeight` 裁剪，内容渲染超出屏幕

**文件**：`view.go:29-42`

```go
// 当前实现：无高度限制
visible := lines[start:]
sb.WriteString(strings.Join(visible, "\n"))
```

`scrollOffset` 较小时 `visible` 包含从 `start` 到末尾的所有行，渲染行数不受终端高度约束，导致每次 PgUp/PgDown 后内容量非线性变化，页面滚动体验混乱，内容溢出终端高度。

**期望修复**：
```go
viewH := m.termHeight - statusBarLines - inputLines
end   := min(start+viewH, len(lines))
visible = lines[start:end]
```

---

#### P1-B：`/compact` 通过 magic string 硬编码拦截，设计脆弱

**文件**：`update.go:210-213`

```go
// 硬编码 name 字符串
if name == "compact" {
    m.activeDialog = dialogCompact
    return m, nil
}
```

`cmdCompact` 返回 `Result{Text: "compact", Display: DisplayNone}`，若此 `if` 不存在，`DisplayNone` 分支会将字符串 `"compact"` 作为用户查询发给 LLM，产生严重误操作。同样问题存在于 `/clear` 的 `if name == "clear"` 处理。

**建议**：在 `commands.Result` 添加 `OpenDialog string` 字段或 `ClearMessages bool` 字段，由命令自声明意图，TUI 通过结构体字段路由，消除 magic string 依赖。

---

#### P1-C：`glamour.NewTermRenderer` 每帧重建，长对话性能退化

**文件**：`messagelist.go:18-29`

```go
func renderMarkdown(md string, width int, dark bool) string {
    r, _ := glamour.NewTermRenderer(       // 每次调用都构造新渲染器（含 CSS 解析）
        glamour.WithStylePath(style),
        glamour.WithWordWrap(width),
    )
    ...
}
```

`View()` 每帧对所有历史消息调用此函数；glamour 初始化含 CSS 解析开销。长对话（>50 条消息）时每帧渲染成本线性增加，不满足测试策略 `<16ms/frame` 性能目标。

**建议**：在 `AppModel` 中缓存 `*glamour.TermRenderer`，仅在 `termWidth` 或 `darkMode` 变化时重建，其余帧复用。

---

#### P1-D：`doAbort` 不清零 `abortFn`/`streamCh`，存在状态残留风险

**文件**：`keys.go:206-212`

```go
func (m AppModel) doAbort() (tea.Model, tea.Cmd) {
    abortCmd := abortQueryCmd(m.queryEngine)
    m.isLoading = false
    m.showSpinner = false
    m.spinner = m.spinner.Reset()
    return m, abortCmd
    // ❌ m.abortFn 和 m.streamCh 未清零
}
```

正常路径下引擎中断后 channel 关闭会经 `StreamDoneMsg` 清理；若引擎有 bug 未关闭 channel，`abortFn`/`streamCh` 永久残留，下次 `startQueryCmd` 行为异常。

**建议**：追加防御性清零：`m.abortFn = nil; m.streamCh = nil`。

### P2（建议优化）

#### P2-A：`newSystemMessage` 使用 `RoleAssistant` 混淆角色，可能污染 LLM 上下文

`model.go:173-180`：错误提示、`/help` 输出等 TUI 内部系统消息使用 `types.RoleAssistant`，可能被引擎在 `GetMessages()` 时传给 LLM 干扰对话上下文。建议添加 `types.RoleSystem` 或在 `GetMessages()` 前过滤。

#### P2-B：Tab 补全只取首个匹配项，不支持循环；`slashSuggestions` 定义未使用

`keys.go:200-202`：多次按 Tab 始终补全到 `matches[0]`，未实现候选循环。`model.go:88-98` 已定义 `slashSuggestions`（含 `selected` 字段）和 `commandDisplayEntry`，均为死代码，应连接至 Tab 处理逻辑或删除。

#### P2-C：`stripANSI` 只处理 `m` 终止符，非颜色 ANSI 序列宽度计算偏差

`coordinator.go:141-158`：ANSI 序列以任意字母结尾，当前实现只识别 `ESC[...m`（颜色），`ESC[2J`/`ESC[H` 等光标控制序列中的字母外字符会写入输出，导致 `CoordinatorPanel` 行对齐偏差。建议替换为正则 `\x1b\[[0-9;]*[A-Za-z]`。

#### P2-D：`slashSuggestions`/`commandDisplayEntry` 类型定义但无任何引用（死代码）

`model.go:88-98`：两个类型均未被任何代码引用，建议补充实现（配合 P2-B）或删除。

#### P2-E：Vim Normal 模式缺少 `G`/`gg`/`w`/`b` 等常用导航命令

`input.go`：`pendingKey` 机制已支持双键序列，扩展成本低。现有 `yy` 执行后无任何用户反馈，建议同步补充常见导航命令。

#### P2-F：非 slash 输入时 Tab 将 `\t` 插入输入框，不符合聊天场景预期

`keys.go:192`：对普通输入（非 `/` 前缀），Tab 被转发给 `textarea` 导致插入制表符。建议非 slash 场景直接 `return m, nil`。

#### P2-G：`StatusBar.tokenUsage` 和 `cost` 字段初始化后从未更新，始终显示 0

`statusbar.go`/`update.go`：`StreamDoneMsg` 处理路径未更新 `m.statusBar.tokenUsage`，token 计数和费用统计功能形同虚设。需在 `StreamDoneMsg` 或新增 `TurnCompleteMsg` 中，从 `engine.Msg` 的 Usage 字段同步更新。

#### P2-H：`abortQueryCmd` 返回 `nil`，abort 完成无可追踪事件

`cmds.go:119-124`：返回 `nil` 合法但不明确。建议返回专用 `AbortedMsg{}` 便于日志记录和未来 "Interrupted" 提示展示。

#### P2-I：`CoordinatorPanel.Tasks` map 只增不减，`EvictAfter` 字段无效

`coordinator.go`：`EvictAfter` 字段已定义，但 `TickMsg` 处理器从不遍历执行驱逐，长期运行的 coordinator 会话中已完成任务永久占用 UI 空间和内存。

#### P2-J：`renderConfirmDialog` 的 `width` 参数被 `_ = width` 丢弃

`view.go:93`：对话框未使用终端宽度约束，窄终端（<40 列）上对话框内容将溢出。建议添加 `lipgloss.NewStyle().MaxWidth(width-4)` 约束。

#### P2-K（QA 新发现）：`Init()` 缺少 `tea.EnterAltScreen`，程序不进入全屏模式

`init.go:29-35`：设计文档 §2 明确要求 `Init()` 返回 `tea.EnterAltScreen`，实际未包含，程序不进入备用屏幕缓冲区，无法实现全屏 TUI 体验。

#### P2-L（QA 新发现）：`startQueryCmd` 接受 `*AppModel` 指针，轻微破坏 MVU 值语义

`cmds.go:14`：通过指针修改调用方 model 的 `abortFn`/`abortCtx`，与 BubbleTea 纯值传递原则略有偏离。建议改为返回修改后的 `AppModel` 值。
